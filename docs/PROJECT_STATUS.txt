PROJECT OVERVIEW
--------------------------------------------------------------------------------
HospiLink connects hospitals with medical staff for duty assignments and
scheduling. Hospitals post duty requirements, and medical staff can view and
accept available duties based on their availability and expertise.

Live URLs:
- Frontend: https://hospilink.in
- Backend:  https://hospilinkv1backend.vercel.app

Hosted on:
- https://hospilink.in

DATABASE SCHEMA
--------------------------------------------------------------------------------

Collections:
  1. users
     • _id, name, email, password (hashed, bcrypt cost 12), role, isEmailVerified
     • Indexes: email (unique), createdAt

  2. tempusers (OTP verification staging)
     • _id, name, email, password, role, otp, otpExpires, createdAt
     • TTL index: auto-delete after 1 hour

  3. medicalstaffs
     • _id, user (ref), fullName, jobRole, city, area, phoneNumber
     • isAvailable, coordinates (named lat/lng), isProfileComplete
     • averageRating, totalRatings
     • Indexes: user (unique), jobRole, city, area, coordinates

  4. hospitals
     • _id, user (ref), hospitalLegalName, currentAddress
     • servicesAvailable (20 specialties enum), location, staffCount
     • coordinates (named lat/lng), isProfileComplete
     • Indexes: user (unique), coordinates, servicesAvailable

  5. duties
     • _id, hospital (ref), assignedTo (ref MedicalStaff), staffRole (44 enum)
     • date, endDate, startTime, endTime, isOvernightDuty
     • urgency (low/medium/high/emergency), description
     • offeredRate, totalPayment (auto-calculated)
     • status (available/assigned/enroute/in-progress/completed/cancelled/incomplete)
     • statusHistory[], cancellation { cancelledBy, reason, reasonText, timestamp }
     • assignedAt, enrouteAt, startedAt, completedAt
     • Indexes: hospital+status, staffRole+status+date, assignedTo+status

  6. notifications
     • _id, recipient (ref User), type (6 event types), payload (Mixed)
     • isRead, createdAt
     • Types: DUTY_CREATED, DUTY_ACCEPTED, DUTY_STATUS_CHANGED,
               DUTY_CANCELLED, DUTY_EDITED, REVIEW_RECEIVED
     • Indexes: recipient+createdAt, recipient+isRead

  7. documents
     • _id, userId (ref User, unique), userRole (staff/hospital)
     • documents[]:
         documentType (enum, 16 types), s3Key, fileName
         extractedText (raw OCR), extractedData (Mixed, parser-structured)
         isDeleted, uploadedAt, updatedAt
         verificationStatus (pending/verified/rejected/auto-verified/
                             manual-pending-verification)
         verifiedBy (ref User), verifiedAt, rejectionReason
         hypervergeData (reserved — disabled)
     • Indexes: userId (unique), userId+documentType, verificationStatus,
                userRole+verificationStatus, verifiedBy, userId+isDeleted

  8. reviews
     • _id, duty (ref, unique), hospital (ref), medicalStaff (ref)
     • rating (1–5), review (text, max 1000 chars)
     • Indexes: medicalStaff

DOCUMENT OCR PIPELINE
--------------------------------------------------------------------------------

Supported Document Types & Parsers:
  Document Type           Parser Output Fields
  ----------------------  -------------------------------------------------------
  aadhaar-card            name, dob, gender, aadhaarNumber, address
  pan-card                name, dob, panNumber
  mcim-certificate        doctorName, registrationNumber
  ncim-certificate        doctorName, registrationNumber
  license-permit          licenseNumber, name, issueDate, expiryDate
  cin-certificate         businessName, cin, incorporationDate
  gst-certificate         legalName, tradeName, businessType, registrationNumber
  nabh-certificate        certificateNumber, hospitalName, validUpto
  rohini-certificate      certificateNumber, hospitalName, validUpto
  cghs-certificate        empanelmentNumber, hospitalName, validUpto

Pipeline:
  1. Upload file buffer to AWS S3
  2. Run OCR via Google Vision API (extractTextFromBuffer)
  3. Pass raw text to document-specific parser
  4. If extractedData has any valid fields → status = "manual-pending-verification"
  5. Aadhaar with extracted number → status = "pending" (awaiting admin review)
  6. Admin manually verifies or rejects with reason


AI AGENT SERVICE
--------------------------------------------------------------------------------

Purpose:
  External job search and matching using Google Gemini AI + SerpAPI

Search Pipeline (9 steps):
  1. Cache check — serve fresh/stale results immediately if available
  2. Search — SerpAPI web search + Google Jobs engine (parallel)
  3. Filter — high-signal snippets vs. needs-fetch classification
  4. Fetch — concurrent HTTP fetch + Cheerio text extraction (50k char limit)
  5. Extract — Gemini LLM batch extraction (3–5 docs per call)
  6. Validate — role + location match validation
  7. Deduplicate — merge by contact/employer
  8. Enrich — reverse lookup for missing contact info
  9. Rank — composite score (role 25%, location 20%, signal 30%,
             recency 10%, confidence 15%)

Delivery:
  • SSE stream (/api/agent/v1/search/stream) — real-time results as they arrive
  • Max 500 concurrent SSE connections
  • Keep-alive pings every 15 seconds

Storage:
  • MongoDB persistence + in-memory fallback
  • Google Sheets export per user session
  • Google Drive file storage

Scheduling:
  • node-cron daily search job
  • Midnight lifecycle tasks (cleanup, stats reset)

APIs Used:
  • Google Gemini AI (text generation / extraction)
  • SerpAPI (web search + Google Jobs engine)
  • Google Sheets API (data export)
  • Google Drive API (file storage)
